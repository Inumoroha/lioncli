## 代码太多太长看不下去？

先说个让你松口气的事实:你这个 `mcp/` 文件夹**一共只有 1123 行**,7 个文件,最大的 `manager.go` 也才 303 行。这不是"多到看不下去",这是一个**一下午就能读透**的规模。你现在的"看不下去",不是因为代码多,是因为**它是 AI 一次性生成的,你没有跟着它长出来的过程**——你是空降到成品里,当然陌生。这事有解,而且解法恰好就能把你"本领没长"这个根治了。

## 为什么会有"本领没长"的感觉

AI 生成代码时,真正值钱的东西没留给你:

- **为什么这么拆分**(为什么 manager / tool / resource / prompt 分开)
- **为什么用这个数据结构**
- **哪里是坑,哪里是权衡**

代码只是结论,思考过程被吞掉了。你要做的不是"读代码",是**把吞掉的思考逆向逼出来**。

## 给你一条具体的路子

**第一步:不要从第一行读。** 先问自己一个问题,然后去代码里找答案:

> "一个 MCP server 从配置文件,到能被调用,中间经过了哪几步?"

带着这个问题读,你会自然地串起 `config.go`(读配置)→ `manager.go`(管理)→ `refresh.go`(连接刷新)→ `tool.go`(暴露工具)。**带问题读 ≠ 逐行读**,前者长本领,后者催眠。

**第二步:用"删了会怎样"逼出设计意图。** 每读到一个函数,问:"如果没有它,会出什么问题?" 答不上来的地方,就是你的知识缺口——那才是该深挖的点。

**第三步:改一个小东西,让它崩。** 主动制造一个 bug(改个参数、删个判断),跑起来看它怎么坏的。**能让代码崩、并解释清为什么崩,就等于真懂了。**

## 每用一个库就要从头学一遍？

它正好戳中你那个"本领没长"的焦虑的**另一面**:你怕的不是代码多,是怕"每用一个库就要从头学一遍,学不完"。

我先给你一个反直觉的结论,然后用**你自己这个包的真实数据**证明它:

> **你永远不需要"学会一个库"。你只需要学会你的代码真正碰到的那一小块表面。**

先看清楚这个"表面"到底有多大:整个包碰到的库符号一共 **37 个**。但关键不在数量,在于它们分成性质完全不同的两类:

## 把 37 个符号劈成"动词"和"名词"

**动词(你真正要"学"的东西)——只有 ~12 个**,而且高度重复:

| 干什么   | 用到的调用                                                   |
| -------- | ------------------------------------------------------------ |
| 建连接   | `NewStdioMCPClient` / `NewStreamableHttpClient`              |
| 握手     | `client.Initialize`                                          |
| 听通知   | `OnNotification`                                             |
| 列清单   | `ListTools` / `ListResources` / `ListPrompts` ← **同一个动作的三份** |
| 实际调用 | `CallTool` / `ReadResource` / `GetPrompt` ← **同一个动作的三份** |
| 收尾     | `GetStderr` / `Close`                                        |

**名词(你根本不用"学")——剩下 25 个全是它**:`CallToolRequest`、`Tool`、`Resource`、`Cursor`、`TextContent`……这些**不是知识,是数据结构**。你不背它们,你是用的时候:`req := mcplib.CallToolRequest{}`,然后让编辑器的自动补全告诉你它有哪些字段。类型系统替你记,你不用记。

## 所以你"学这个库"的真实成本是多少?

把上面那张表再压一下,你会发现 ListTools/ListResources/ListPrompts 是**同一个形状**,CallTool/ReadResource/GetPrompt 也是**同一个形状**。所以真正要理解的"独立动作"其实只有:

> **建连 → 握手 → 听通知 → 列清单(分页)→ 调用 → 关闭。六个。**

这就是为什么我在 [zzz.md](vscode-webview://0392vca9q0fquf0bqu0dq22l6n0cvmqklndqgh30kegeejroqdef/internal/mcp/zzz.md) 里跟你说"**看懂 `tool.go` 一个,resource/prompt 跳着看即可**"——库的 API 本身就是重复的,你学一遍 tool 那条线,另外两条是白送的。

**六个动作,不是一个库。** 这就是你的真实学习成本。

## 这背后是一个能迁移到任何库的方法

你问的其实是个更大的问题:"以后每个库都这样,学得完吗?"——学不完,**但你本来就不该"学完一个库"**。正确的姿势是:

```
❌ 错的:打开库文档从头读 → 想先"掌握"它 → 读到一半忘了自己要干嘛
✅ 对的:从"我要干什么"出发 → 只查那一个动作怎么调 → 用类型/补全补全细节 → 下一个需求再查下一个
```

`mcp-go` 这个库本体可能有几千上万行、几十上百个导出符号。**你的包碰了 37 个,真正理解了 6 个动作。** 剩下 99% 你这辈子可能都不会用到——去学它就是纯浪费。

## 回到你那个"本领没长"的焦虑

我要专门点一下这件事,因为它正好是反的:

> **"知道一个库不用学完、只学自己要碰的那 6 个动作"——这个判断力本身,就是高级工程师和新手最大的区别之一。**

新手的恐惧是"这库这么大我学不完所以我不行";有经验的人的反应是"我只需要列清单和调用两件事,十分钟找到这两个函数就开干"。你现在能问出这个问题,说明你已经隐约摸到这个分界线了。

下一步把它变成肌肉记忆的办法很简单:下次再加一个库时,**别开文档首页**,直接搜"这个库怎么做 [我要的那一件事]",查到就用,用到的类型让补全教你。逼自己几次,这个"按需学习"的本领就长出来了——这才是真正不会贬值的本事,比记住任何一个具体库的 API 都值钱。

## 先拆穿一个让你泄气的"不公平比较"

"怎么想得出"——这才是你真正焦虑的那个本领

你现在是在拿**你的"空白页大脑"**(还没动手、什么上下文都没有)去比**AI 的"成品"**(它已经把整个问题趟完了一遍才吐出来)。这个比较当然让你绝望——但它是假的。

公平的比较是:**如果让你一行行写下去、撞上墙再改,你能不能"演化"出这个设计?** 答案是能。
