```

└──────────────── 函数名和参数 ────────────────┘  └──── 返回值类型 ────┘
```

返回值类型是 `func(mcplib.JSONRPCNotification)`——**一个函数**。也就是说:

## 二、为什么要搞这么绕?直接写一个 handler 不行吗?

这是关键,理解了**为什么**,这个结构就再也不绕了。

回到 manager.go 里它被怎么用的(第 232 行):



```go

```

`OnNotification` 要求你给它一个 handler 函数,签名固定就是 `func(n JSONRPCNotification)`——**只有一个参数 n,没有 serverName**。这是库规定的,你改不了。

可问题来了:当通知真的来临、这个 handler 被调用时,它**必须知道"这条通知是哪个 server 发来的"**,才能往纸条上写对地址(`refreshReq{server: serverName}`)。

但 handler 的参数里**只有 n,没有 serverName**。serverName 从哪来?

**答案:用闭包"提前焊进去"。**

`makeNotificationHandler(r.name)` 调用的那一刻,`serverName` 就被定死成比如 `"github"`。它返回的那个内层函数,**永久记住了** `serverName = "github"`,哪怕以后通知来的时候,外层函数早就返回结束了——这个被记住的变量,就叫**闭包捕获**。

所以本质是:

> 库只让我传 `func(n)` 这一种形状的函数,但我需要在里面用到 serverName。于是我用一个工厂,把 serverName **包(closure)进**函数里,造出一个"**已经知道自己是哪个 server**"的 handler。

每个 server 调一次,造一个专属 handler:

```go
makeNotificationHandler("github")  → 一个永远往纸条写 "github" 的 handler
makeNotificationHandler("slack")   → 一个永远往纸条写 "slack"  的 handler
```

**这就是整个绕的理由。** 它不是炫技,是被库的固定签名"逼"出来的标准解法。

## 三、现在看内层函数,它特别简单

内层函数被调时(通知真来了),只干两件事:

**第 1 件:翻译。** server 发来的通知里有个 `n.Method`(一个字符串,比如 "tools 列表变了")。switch 把这个字符串翻译成一个**枚举值** `kind`:



```go
switch n.Method {
case ...ToolsListChanged:     kind = refreshTools     // 工具变了
case ...ResourcesListChanged: kind = refreshResources // 资源变了
case ...PromptsListChanged:   kind = refreshPrompts   // prompt 变了
default:                      return                  // 别的通知?不关我事,直接走人
}
```

`default: return` = "收到我不认识的通知,什么都不做,返回"。

**第 2 件:扔纸条进信箱。** 就是我们昨天讲的那个**非阻塞投递**:



```go
select {
case m.refreshCh <- refreshReq{server: serverName, kind: kind}: // 信箱没满 → 扔进去
default:                                                        // 信箱满了 → 不等,直接丢,打个日志
    log.Printf("...刷新队列已满，丢弃...")
}
```

注意纸条 `refreshReq{server: serverName, kind: kind}` 里的 `serverName`——**就是闭包提前焊进来的那个**。这下整条线打通了:工厂把 server 名焊进 handler → handler 收到通知 → 把"哪个 server + 什么变了"写成纸条 → 非阻塞扔进 channel → worker 去取。

## "有什么不同":❌ 这里要拆开看

你说"kind 和 serverName 不同"——**serverName 对,kind 错。** 区别在哪?在于这两个变量**诞生的时机完全不同**:

**`serverName`——在"造 handler 的时候"就定死了。**



```go
makeNotificationHandler("github")  // 这一刻,serverName 永久 = "github"
```

它是工厂的**入参**,闭包**捕获**它,焊死。这个 handler 这辈子 serverName 都是 "github"。所以 3 个 handler 各自的 serverName 不同 → **你说对了**。

**`kind`——不是 handler 之间的区别,它在每个 handler \**内部\**每次被调用时才临时算出来。**

看清楚 `kind` 在哪声明的:



```go
return func(n mcplib.JSONRPCNotification) {
    var kind refreshKind     // ← 注意!它在内层函数体里,不是工厂入参
    switch n.Method { ... }  // ← 每次通知来了,根据这条通知的 n.Method 现算
}
```

`kind` 是个**局部变量**,handler **每次被调用**时重新声明、根据**这一次**收到的通知内容(`n.Method`)临时算出来。

所以**同一个** "github" handler:

- 这次收到"工具变了" → 这次 `kind = refreshTools`
- 下次收到"prompt 变了" → 下次 `kind = refreshPrompts`

`kind` 是**随每条通知变化的**,它**不是** handler 之间的差异,而是**同一个 handler 在不同时刻**的差异。

------

## 用一句话钉死这个区别

| 变量         | 何时确定                                            | 是什么层面的差异                                   |
| ------------ | --------------------------------------------------- | -------------------------------------------------- |
| `serverName` | **造 handler 时**(工厂入参,闭包捕获)                | **handler 与 handler 之间**的差异(github vs slack) |
| `kind`       | **handler 每次被调用时**(局部变量,按 n.Method 现算) | **同一 handler 不同次调用之间**的差异              |

所以正确答案是:

> 3 个 handler 的唯一区别是**各自捕获的 serverName 不同**。`kind` 不是它们的区别——它是每个 handler 内部、每次处理通知时根据通知类型临时计算的局部变量。

------

## 为什么这个区分极其重要(面试官就爱在这埋雷)

因为它考的是你脑子里有没有这两条**时间线**:

1. **"造函数"的时间线**(工厂被调,捕获 serverName)—— 发生 3 次,很早。
2. **"用函数"的时间线**(通知到来,handler 执行,算 kind,扔纸条)—— 发生无数次,很晚,而且每次的 n 都不同。

**闭包的全部精髓**就是:第 1 条时间线捕获的东西(serverName)被**冻住**,带到了第 2 条时间线里用。而局部变量(kind)只活在第 2 条时间线的**单次**调用里,调用结束就销毁,下次重来。

你昨天问"怎么有专业感"——**就是这个**。能把"捕获的变量"和"每次调用的局部变量"分清楚讲出来,面试官立刻知道你**真懂闭包**,而不是背了个"闭包能记住外部变量"的口号。

## 一、先分清:goroutine 和闭包是两个维度的东西

看这行:



```go
go func(name string, st *serverState) { ... }(serverName, s)
```

它身上同时有两个身份:

- **`go` 关键字** → 让后面这个函数**在一个新 goroutine 里跑**(并发执行)。这是"**怎么跑**"。
- **`func(...){...}` 这个匿名函数** → 它本身就是个函数,只要它用到了外面的变量,它就是个**闭包**。这是"**它是什么**"。

所以一句话:**这是"用 go 启动的一个闭包"。** goroutine 是它的运行方式,闭包是它的本质。两者不矛盾,是同一行代码的两面。

**而坑,恰恰诞生在这两者的结合处** —— 因为 goroutine 让函数"**晚一点、异步地**"跑,而闭包捕获的变量在那个"晚一点"的时刻可能**已经变了**。

------

## 二、把坑演示出来(看错误写法)

假设作者**没有**把循环变量传进去,而是图省事直接写(这是 Go 里最经典的 bug):



```go
for serverName, s := range m.servers {   // 假设有 "github"、"slack"、"notion" 三个
    go func() {                            // ← 注意:没有参数,直接用外面的 serverName
        st := s.client.Initialize(...)     // 用到了 serverName 和 s
        results <- hsResult{name: serverName, ...}
    }()
}
```

凭直觉,你以为 3 个 goroutine 分别处理 github、slack、notion。

**但极可能 3 个都在处理 "notion"(最后一个)。** 为什么?

------

## 三、坑的根源:循环变量是"同一个盒子",被反复改写

关键认知:`for serverName := range ...` 里的 `serverName`,**不是每轮新建一个变量,而是同一个变量被反复赋值**(在老版本 Go 里)。把它想成一个**贴着 "serverName" 标签的盒子**:



```
第1轮:盒子里放 "github"  → 启动 goroutine A(它记住的是"那个盒子",不是"github")
第2轮:盒子里改成 "slack"  → 启动 goroutine B(同样记住"那个盒子")
第3轮:盒子里改成 "notion" → 启动 goroutine C(同样记住"那个盒子")
```

**致命点在于 `go` 的"异步":** goroutine A/B/C 被 `go` 启动后,**不是立刻就跑**,它们排队等调度,可能要等到整个 for 循环都结束了才真正开始执行。

等它们终于开跑、去读那个盒子时——for 循环早结束了,**盒子里现在装的是最后一轮的 "notion"**。于是三个 goroutine 读到的全是 "notion"。

> **闭包捕获的是"变量"(那个盒子),不是"捕获那一刻盒子里的值"。** 等它真正去读的时候,盒子可能已经被改了 N 次。
>
> goroutine 把"读取"推迟到了未来,而循环把盒子改了——两者一结合,bug 就出现了。

这就是为什么我说**坑在 goroutine 和闭包的结合处**:

- 纯闭包不异步(造完立刻用),盒子还没来得及变,可能看不出问题;
- 一旦用 `go` 让它**延后执行**,盒子被循环改完了它才来读,坑就爆了。

------

## 四、作者的修复:把值当参数"拍一张快照"

现在回看 manager.go 第 205 行的**正确写法**:



```go
for serverName, s := range m.servers {
    go func(name string, st *serverState) {   // ← 声明了形参 name、st
        ...
        res, err := st.client.Initialize(...)
        results <- hsResult{name: name, ...}   // ← 用的是形参 name,不是外面的 serverName
    }(serverName, s)                            // ← 把这一轮的值"传进去"
}
```

魔法在最后那行 `(serverName, s)`:

**函数传参是"值拷贝"。** 在第 1 轮,`serverName` 此刻的值 "github" 被**拷贝**进形参 `name`——`name` 是这个 goroutine **私有的、独立的**一份,和那个会被改写的"盒子"再无关系。第 2 轮的 "slack" 拷进另一个 goroutine 私有的 `name`……



```
goroutine A 私有的 name = "github"  (一张快照,永不改变)
goroutine B 私有的 name = "slack"
goroutine C 私有的 name = "notion"
```

哪怕 goroutine 们全都拖到循环结束后才跑,各自手里攥的是**自己那张快照**,互不干扰。坑解决。

注释 195 行那句"写 m.servers 不需要锁"是另一回事,但 `go func(name, st)` 这个传参动作,**就是专门为了躲循环变量捕获这个坑**。
