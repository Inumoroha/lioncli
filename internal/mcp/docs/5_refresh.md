> `2026/06/12 10:47:36`

## 先讲故事:它在干嘛?

MCP server(比如一个提供工具的外部进程)有个特性:**它的工具列表会变。** 比如某个 server 原本提供 3 个工具,运行中它新增了一个,或下线了一个。这时它会主动给你**推一条通知**:"喂,我的工具列表变了,你重新拉一下。"

你的程序收到这条通知后,该做的事很简单:**重新调一次 `ListTools` RPC,把最新列表拉回来,更新本地缓存。**

听起来一句话的事。但这里有个**致命的坑**,refresh.go 整个文件的复杂度,90% 都是为了绕开这一个坑:

> **收到通知的那个"瞬间",你不能在原地直接去拉列表。**

为什么?这就是整个文件的灵魂。

```
handler 在 transport 读循环里同步被调，不能阻塞、不能发 RPC
```

## 那个致命的坑(理解这个,你就懂了 80%)

想象 MCP 客户端和 server 之间有**一根管道**。这根管道是**单线程**的:有一个专门的循环(叫"读循环")不停地从管道里读东西。读到的东西有两种:

1. **你之前发的请求的"回复"**(比如你调 `ListTools`,它的结果从这根管道回来)
2. **server 主动推的"通知"**(比如"工具变了")

**关键:这两种东西,都从同一根管道、由同一个读循环处理。**

现在重点来了。当读循环读到一条"工具变了"的通知,它会调用你注册的处理函数(handler)。**此刻,读循环正卡在你的 handler 里,动不了。**

如果你在 handler 里直接写"那我去调 `ListTools` 吧"——

- `ListTools` 这个请求发出去了
- 它的**回复**要从哪根管道回来?**同一根管道**
- 谁负责读那根管道把回复捞出来?**那个读循环**
- 读循环在哪?**正卡在你的 handler 里等 `ListTools` 返回**

→ 你在等回复,而唯一能给你回复的人正在等你。**死锁。** 整个 MCP 客户端永久冻住。

这就是注释说的"在读循环里发 RPC 会死锁"。

## 解法:把"收到通知"和"实际去拉"拆开

既然 handler 里**不能干重活**,那 handler 就只干一件最轻的事:**写个纸条扔进信箱,立刻走人。** 重活让**另一个人**(另一个 goroutine)从信箱里取纸条慢慢干。

这就是整个 refresh.go 的架构,就这三个角色:

```
┌─────────────┐  通知来了    ┌──────────┐  扔纸条   ┌─────────┐  慢慢干
│ 读循环       │ ──调用──→   │ handler  │ ──→     │ refreshCh │ (信箱/channel)
│ (transport) │             │ 只扔纸条  │         │  缓冲 64  │
└─────────────┘             └──────────┘         └────┬────┘
   ↑ 立刻被释放,继续读管道                              │ 取纸条
                                                  ┌────▼─────┐
                                                  │  worker   │ 独立 goroutine
                                                  │ 真正发 RPC │ 在这里调 ListTools 才安全
                                                  └──────────┘
```

- **handler**(refresh.go:35-56,`makeNotificationHandler`):收到通知,翻译成一个 `refreshReq`(纸条),`select` 扔进 `refreshCh`。**关键是它扔完立刻返回**,读循环马上被释放,不会死锁。
- **refreshCh**(信箱,一个 channel):manager.go:43 定义,缓冲 64 条。
- **worker**(refresh.go:60-73,`startWorker`):一个**独立的、长期活着的** goroutine,死循环从 `refreshCh` 取纸条,取到一张就调 `handleRefresh` → 真正去发 `ListTools` RPC。**worker 不在读循环里,所以它发 RPC 不会死锁。**

**这就是这个文件的全部主线。** 其余都是细节。

## 现在回头看代码,你会发现它顺了

**handler(扔纸条的人),refresh.go:50-54:**

```go
select {
case m.refreshCh <- refreshReq{server: serverName, kind: kind}:  // 信箱没满,扔进去
default:                                                          // 信箱满了,直接丢,不等
    log.Printf("⚠️ ...刷新队列已满，丢弃...")
}
```

这个 `select + default` 是 **"非阻塞发送"** 的固定写法。意思是:能扔就扔,信箱满了**绝不傻等**——因为一旦在这里等,读循环又被卡住,又回到死锁。注释 48-49 行说的就是这个。丢一张纸条没关系(注释 27-30 解释了:下次通知或重连会重新拉,状态不会永久错)。

**worker(干活的人),refresh.go:64-71:**

```go
for {
    select {
    case <-m.done:           // 收到关闭信号 → 退出
        return
    case req := <-m.refreshCh: // 信箱有纸条 → 取出来,干活
        m.handleRefresh(req)
    }
}
```

死循环。要么收到"下班"信号(`m.done`)就退出,要么从信箱取一张纸条处理。这是 Go 里 **worker goroutine 的最经典模板**,你看一次记一辈子。

**真正干活,refresh.go:90-126,`refreshToolsFor`:**

1. 调 `listAllTools` 去 server 拉最新列表(**这一步是 RPC,所以必须在 worker 里、不能在 handler 里**)
2. 拉失败就**保留旧缓存**直接返回(注释 98:清空再失败会让 LLM 突然啥工具都看不到,更糟)
3. 拉成功 → 加写锁 `m.mu.Lock()` → 把这个 server 的旧 tools 删掉、塞入新的 → 解锁
4. `fireToolsChanged()` 通知上层"工具变了"(比如让 UI 刷新)

你看第 3 步那段加锁的删改:

```go
m.mu.Lock()
for k, e := range m.tools { if e.serverName == serverName { delete(m.tools, k) } }
for k, e := range newEntries { m.tools[k] = e }
m.mu.Unlock()
```

**这就是我之前跟你说的"复利"**——还记得 status.go 里 `StatusSnapshot` 用 `RLock` 读 `m.tools` 吗?这里就是那个"写的一方"。读用 `RLock`、写用 `Lock`,它俩**抢的是同一把锁 `m.mu`**。两个文件在这里咬合上了。status.go 是读者,refresh.go 是写者,锁让它们不会撞车(不会触发你学的那个 `concurrent map read and map write` fatal error)。

## 一句话总结整个文件

> **refresh.go 解决的是:"MCP server 说它列表变了,我要去重新拉,但我不能在收到通知的当场拉(会死锁),所以我把'去拉'这个动作通过一个 channel 转交给一个后台 worker 来做。"**

`refreshTools/Resources/Prompts` 三套几乎一模一样的代码(90-200 行),只是把同一套逻辑分别对 tools、resources、prompts 各做一遍而已——**你看懂 `refreshToolsFor` 一个,另外两个直接跳过**,它们是复制粘贴。

> **为什么 handler 不能直接调 `ListTools`,非要绕一个 channel 交给 worker?把那个"死锁"用你自己的话讲一遍。** 讲对了,这个文件就真是你的了。讲不顺,我换个角度再给你比划。

**完全正确,而且你讲的是对的。** 一字不差,这就是死锁的完整因果链。我给你把它"术语化"一下,这样面试时你能用更专业的词把它讲出来:

> "MCP 客户端的 transport 是**单线程读循环**,请求响应和服务端通知**复用同一条读取通路**。通知 handler 是被读循环**同步回调**的,如果在 handler 内**重入**式地发起一个 RPC,该 RPC 的响应需要由同一个读循环来接收,而读循环此刻正阻塞在 handler 内,形成**自死锁**(self-deadlock)。因此 handler 必须**非阻塞**地把刷新请求投递到 channel,由独立的 worker goroutine 异步执行真正的 RPC。"

里面三个词是加分项,记一下:

- **同步回调**(handler 是被读循环当场调的,不是另开线程)
- **重入 / self-deadlock**(在持有某资源时又去申请依赖该资源的操作)
- **非阻塞投递**(那个 `select + default`)

你能把"单一读循环 + 复用通路 + 同步回调 = 重入会自死锁"这条链讲清楚,这道题在面试里就是**满分答案**,而且是那种能让面试官眼睛一亮的"他是真懂"的回答——因为很多人只会背"channel 解耦",讲不出**为什么非解耦不可**。



1. **`Close()` 的关闭顺序**(manager.go:247)——为什么必须**先关 worker 再关 client**?顺序反了会怎样?(注释里藏了答案,但能讲出"为什么"的人不多)
2. **`fire` 系列的"先复制切片再调回调"**(refresh.go:225-240)——为什么不能持着锁直接调回调?这里藏着一个**回调重入死锁**,和刚才的死锁是"同一类病的不同表现"。
3. **`Initialize` 的并发握手**(manager.go:203-217)——`go func(name, st)` 为什么要**把循环变量当参数传进去**?这是 Go 最经典的坑之一(闭包捕获循环变量)。

"MCP 客户端的 transport 是单线程读循环,请求响应和服务端通知复用同一条读取通路。通知 handler 是被读循环同步回调的,如果在 handler 内重入式地发起一个 RPC,该 RPC 的响应需要由同一个读循环来接收,而读循环此刻正阻塞在 handler 内,形成自死锁(self-deadlock)。因此 handler 必须非阻塞地把刷新请求投递到 channel,由独立的 worker goroutine 异步执行真正的 RPC。"这个表述太专业了，可是你也看见了我只会说口水话，怎么做才能达到这种专业感爆棚的感觉呢？

这是今天**最该问的问题**,比任何一道技术题都值钱。我给你一个**反直觉的真相**,然后给你一个能立刻练的方法。

## 真相:专业感 ≠ 你比别人懂得多,而是你给概念"装了把手"

| 你的口水话                 | 专业词(把手)                     |
| -------------------------- | -------------------------------- |

## 方法:"口水话 → 找把手"翻译练习(每天 10 分钟,立竿见影)

别去背术语表,那样记不住。用这个流程,它的核心是**永远从你已经懂的口水话出发**:

**第 1 步:用口水话讲懂一个机制**(你已经天然会这步) 比如你说:"它先把回调切片复制一份再去调,不然在锁里调回调,回调又来注册,就又锁死了。"

**第 2 步:把这句话里的"动作"一个个圈出来,问"这个动作有没有专门的词?"**

- "复制一份再用" → 这叫 **快照(snapshot)** / **写时复制**
- "在锁里又触发需要锁的操作" → 这叫 **锁重入 / 重入死锁**
- "复制后再调,把调用挪到锁外" → 这叫 **缩小临界区(critical section)**

**第 3 步:用找到的词,把第 1 步那句话重讲一遍** "`fire` 系列**在临界区内只做切片快照,把回调的实际执行移出锁**,避免回调**重入**注册逻辑导致的**自死锁**。"

**第 4 步(最关键):合上代码,只用这些词,讲给空气听一遍。** 讲不顺 = 这个词你还没"装上",回第 1 步。讲顺了 = 这个把手焊死了。
