## 一份"这份代码能撑起的面试考点"清单

这套代码的含金量很高,我帮你把它映射到面试高频考点——**每一条你都要能指着自己的代码讲**:

| 考点                    | 在你代码里的位置                                    | 一句话要点                                                  |
| ----------------------- | --------------------------------------------------- | ----------------------------------------------------------- |
| 读写锁 vs 互斥锁        | `manager.go` `mu sync.RWMutex`                      | 读多写少:状态查询频繁、refresh 偶发,RLock 让并发读不互斥    |
| 为什么 servers 不在锁内 | `manager.go:38` 注释                                | 握手阶段独占构建,之后只读 → 无需加锁,讲"不可变即安全"       |
| 同步回调不能阻塞        | `refresh.go` makeNotificationHandler                | transport 读循环里发 RPC 会死锁,所以解耦成 channel + worker |
| 有界 channel + 丢弃策略 | `refresh.go` `refreshChanBuf=64` + `select default` | 宁可丢一次通知(下次会兜底)也不能阻塞读循环                  |
| 单 worker 串行化        | `refresh.go` startWorker                            | 避免并发 List,也避免在读循环发 RPC                          |
| 优雅关闭顺序            | `manager.go` Close                                  | 先停 worker 再关 client,否则撞 closed channel               |
| 锁的粒度/死锁规避       | `refresh.go:225` fire 系列                          | 回调前先复制切片释放锁,防回调里重入加锁                     |
| 容错隔离                | `Initialize` 单 server 失败只跳过                   | 故障隔离,一个坏 server 不拖垮整体                           |
| 缓存更新策略            | `refreshToolsFor` 失败保留旧缓存                    | "清空再失败"比"保留旧值"更糟,可用性优先                     |

这张表你能讲下来,MCP 这块就从"AI 写的"变成"我设计的"了。
